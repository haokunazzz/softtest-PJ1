## 术语和缩略语

| 术语 / 缩略语 | 含义 |
| --- | --- |
| SUT | System Under Test，被测系统，本项目指场馆预约管理系统 |
| Service 层 | Spring Bean 中实现业务逻辑的服务类，对应 `com.demo.service.impl.*` |
| Controller 层 | 接收 HTTP 请求并调度业务逻辑的控制器，对应 `com.demo.controller.**` |
| 单元测试（UT） | 以单个方法 / 类为对象、隔离协作者的测试，本项目以 service 层为主 |
| 集成测试（IT） | 启动 Spring 上下文 + 真实 / 内嵌库的端到端测试，本项目以 controller 层为主 |
| MockMvc | Spring 提供的 HTTP 模拟客户端，可在不启端口的情况下发起请求 |
| H2 | Java 编写的内嵌数据库，本项目用其 MySQL 兼容模式做测试库 |
| JaCoCo | Java Code Coverage 工具，提供指令、行、分支、方法等多维覆盖率 |
| 指令覆盖率 | JaCoCo 默认度量；统计字节码指令被执行比例，与 IEEE 829 中的"语句覆盖"近似（详见附录 D） |
| 行覆盖率 | JaCoCo 中每行至少有一条指令被执行的比例 |
| 分支覆盖率 | JaCoCo 中每个分支（true/false）被执行比例，与 IEEE 829 中的"判定覆盖"近似 |
| 等价类划分 | 黑盒测试技术，将输入域划分为若干等价类后选代表用例 |
| 边界值分析 | 黑盒测试技术，针对取值边界附近设计用例 |
| 缺陷严重等级 | Blocker / Critical / Major / Minor / Trivial 五级（见 §8.4） |
| 测试基线 | 测试代码与被测代码同一 git revision 的快照 |

## 1 目的

本计划面向附件 4 提供的"场馆预约管理系统"，目标是：

1. 规定本次课程项目测试工作的**范围、方法、资源、时间表与交付物**，使整个测试过程可执行、可重复、可度量、可审计。
2. 明确**测试项**与**要测试 / 不会被测试的特性**，避免范围漂移；明确**通过 / 失败、暂停 / 继续**等准则。
3. 落实**小组分工与责任**，识别测试过程中的**主要风险及应急措施**。
4. 为《测试用例设计文档》《测试总结》提供**统一的口径基准与编号体系**，使三份文档在测试范围、覆盖率口径、缺陷分级等方面保持一致。
5. 满足课程作业 PJ1 提交要求：本计划书将与《测试用例设计文档》《测试总结》一同导出 PDF，与测试脚本源码打包为 `PJ1-小组编号.zip` 提交至 elearning。

## 3 介绍

### 3.1 被测系统概述

被测系统是一个基于 Spring Boot 2.2.2 的 Web 端**场馆预约管理系统**，使用 Thymeleaf 渲染前端页面，通过 Spring Data JPA + MySQL 持久化业务数据，支撑两类角色：

- **普通用户**：浏览场馆与新闻、注册登录、预约场馆、维护个人订单与留言。
- **系统管理员**：管理用户、场馆、新闻、审批订单、审核留言。

### 3.2 系统来源与版本

- 系统出处：课程附件 4 `SoftwareTestingDemo.zip`，为课程提供的演示项目，非本组自研。
- 当前版本：`com.demo:demo:0.0.1-SNAPSHOT`（见 `pom.xml`）。
- 项目阶段：**测试验证阶段**，代码已稳定，本组仅做测试，不进行功能性变更。

### 4.2 先决条件

执行任何测试前必须满足：

1. 安装 **JDK 1.8** 与 **Maven 3.6+**；`mvn -v` 与 `java -version` 均正常。
2. 项目根目录为 `SoftwareTestingDemo/SoftwareTestingDemo`（含 `pom.xml`），可成功执行 `mvn -DskipTests package`。
3. **不依赖本机 MySQL**：`application.yml` 已切换为 H2 内存库（MySQL 模式 + 大小写不敏感）。
4. 测试代码与被测代码处于**同一 git 基线**，以保证可重复性；任何源码修改都需重新跑全量回归。
5. 端口冲突由 `server.port: 0` 自动随机分配，不与本机现有服务冲突。
6. 测试机磁盘空间 ≥ 1 GB（用于 Maven 依赖与 JaCoCo 报告）。

### 4.3 移交事件触发条件

被测代码视为可进入"正式测试执行阶段"，需同时满足：

- A 已完成全部测试脚本编写；
- 一次完整 `mvn clean test` 在干净环境下成功运行（无 compile 错误）；
- 覆盖率报告 `target/site/jacoco/index.html` 已生成；
- A 已发布 `A任务交接文档.md`。

上述条件已于 2026-04-18 满足（见 `A任务交接文档.md` §2.4）。

## 6 不会被测试的特性

| 不测特性 | 理由 |
| --- | --- |
| 浏览器端 UI 自动化（Selenium 端到端） | 课程评分聚焦在 service 单元测试 + controller 集成测试，UI 层不在 PJ1 评分点 |
| 真实文件系统写入（`FileUtil` 落盘） | 涉及环境路径，且与业务正确性无关，作为已知限制写入《测试总结》 |
| 真实 MySQL 行为差异 | 用 H2 MySQL 模式近似覆盖 SQL 语义；MySQL 专有行为不属于课程考察 |
| 性能 / 压力 / 并发负载测试 | 课程范围未要求；本项目无明确性能 SLA |
| 安全测试（渗透 / SQL 注入 / XSS） | 课程范围未要求 |
| Thymeleaf 模板的视觉渲染 | MockMvc 仅校验状态码、视图名与模型属性，不校验最终 HTML 像素 |
| 第三方依赖自身的功能 | 不重复测试 Spring、Hibernate、JUnit、Mockito 等依赖库内部逻辑 |

## 7 方法

### 7.1 总体方法

采用**黑盒 + 白盒**结合、**单元 + 集成**两层并行的策略：

| 层次 | 测试类型 | 主要技术 | 主要目标 |
| --- | --- | --- | --- |
| service | 单元测试 | JUnit 5 + Mockito 隔离 DAO | 验证业务逻辑分支正确性，覆盖语句 / 判定 |
| controller | 集成测试 | Spring Boot Test + MockMvc + H2 | 验证 HTTP 入口到业务的端到端正确性，覆盖请求映射 |

### 7.2 黑盒测试技术（满足课程要求）

- **等价类划分**：对每个被测函数 / 接口，划分有效等价类（合法用户、合法订单状态等）和无效等价类（不存在用户、未登录、非法状态等）。
- **边界值分析**：对分页参数（`page = 0 / 1 / 末页 / 末页+1`）、字符串长度（空串 / 边界长度 / 超长）、状态枚举边界、ID 边界（不存在 / 0 / 负数）等设计用例。
- **场景法 / 状态流转法**：用于订单与留言"待审核 → 通过 / 拒绝 / 完成"等状态机路径覆盖，覆盖合法转换与非法转换。

### 7.3 白盒测试技术（满足课程要求）

- **语句覆盖**：保证被测方法的可达语句被至少执行一次，对应 JaCoCo 的指令覆盖与行覆盖度量（详见附录 D）。
- **判定覆盖**：保证关键 if/else、switch 的真假分支均被覆盖，对应 JaCoCo 的分支覆盖。
- **配合 JaCoCo 度量**：执行 `mvn test` 后查看 `target/site/jacoco/index.html`，按指令覆盖率与分支覆盖率自查；不达标的方法回到 §7.2 补用例。

### 7.4 完整性与冗余控制（PJ1 说明文档明确要求）

- **完整性**：每个 service public 方法 ≥ 1 条正常路径用例 + ≥ 1 条异常 / 边界用例；每个 controller mapping ≥ 1 条用例；关键状态机所有合法转换与典型非法转换均覆盖。
- **冗余控制**：等价类内每类只取 1~2 条代表用例；公共测试数据由 `TestDataFactory` 集中管理；公共初始化由 `AbstractControllerIntegrationTest` 提供，避免重复脚本。

### 7.5 主要工具

| 工具 | 用途 |
| --- | --- |
| JUnit 5 | 测试运行与断言 |
| Mockito | 单元测试中 mock DAO / 协作对象 |
| Spring Boot Test | 启动上下文进行集成测试 |
| MockMvc | 在不启 HTTP 端口的情况下模拟请求 |
| H2 内存库 | 替代 MySQL，使集成测试可在任意机器执行 |
| JaCoCo | 生成覆盖率报告（指令、行、分支、方法） |
| Maven Surefire | 输出测试报告 `target/surefire-reports/` |
| Maven | 统一构建与测试入口（`mvn test`） |

### 7.6 测试广度与完成准则

- 每个 `service.impl` public 方法：**正常 + 异常 / 边界**两类用例齐备；
- 每个 controller mapping：至少覆盖 1 条用例；
- 关键状态机：合法路径全覆盖、典型非法路径覆盖；
- 覆盖率**目标**：指令覆盖率 ≥ 85%、分支覆盖率 ≥ 75%（不作为阻断条件，仅作目标）；
- **错误频率约束**：每 100 条测试用例失败 ≤ 1 条，且不含 Blocker / Critical 级缺陷（缺陷分级见 §8.4）。

### 7.7 重新测试触发条件

发生以下任一情况，需要对受影响范围**重新执行回归**：

1. 被测源码修改（含 hot-fix）；
2. 测试基座（`AbstractControllerIntegrationTest`、`TestDataFactory`、`application.yml`）修改；
3. `pom.xml` 中依赖版本变更；
4. 任一 Blocker / Critical 缺陷修复后；
5. JaCoCo 报告丢失或报告日期早于代码最近一次提交。

### 7.8 需求与用例的可追溯性

需求 → 功能 → 测试设计 → 测试用例 → 测试脚本之间的追溯通过编号体系串联（见前述"编号体系"小节），完整追溯矩阵见**附录 C**；该矩阵由 C/D 在用例文档中维护并保持同步。

### 7.9 主要约束

- **时间约束**：PJ1 截止 2026-04-26 24:00。
- **资源约束**：本机可能不带 Maven，全量回归至少需要一台具备 JDK 1.8 + Maven 3.6+ 的环境。
- **范围约束**：仅 service 与 controller 两层，不做 UI 自动化、性能、安全测试。

## 8 测试项通过/失败准则

### 8.1 用例级准则

| 项 | 通过条件 | 失败条件 |
| --- | --- | --- |
| 单元测试用例 | 全部断言成立、无未预期异常 | 任一断言失败、抛出未预期异常、超时 |
| 集成测试用例 | HTTP 状态码、视图名 / JSON 结构、模型属性符合预期 | 状态码不符、跳转页面不符、返回数据缺失或格式错误 |

### 8.2 模块级准则

某模块（service 或 controller）测试**通过**需同时满足：

1. 该模块所有用例 100% 通过；
2. 该模块覆盖率达标：service 模块**指令覆盖率 ≥ 85% 且分支覆盖率 ≥ 75%**；controller 模块**mapping 覆盖率 = 100%**；
3. 不存在 Blocker / Critical 级缺陷。

### 8.3 项目整体通过准则

1. `mvn test` 一次性全部通过，无 failure、无 error；
2. JaCoCo 报告生成成功，**指令覆盖率 ≥ 85%、分支覆盖率 ≥ 75%**；
3. 所有测试在新克隆环境中可重复执行（H2 内存库、零外部依赖）；
4. 计划书、用例文档、总结报告三份文档齐备并已导出 PDF；
5. 缺陷登记表中无未关闭的 Blocker / Critical 级缺陷。

### 8.4 缺陷严重等级定义

| 等级 | 代号 | 定义 | 处置策略 |
| --- | --- | --- | --- |
| Blocker | B | 阻断性缺陷：导致核心功能不可用、整批用例无法执行、或测试基座失效 | 立即停下，必须修复后才能继续测试 |
| Critical | C | 严重缺陷：核心业务路径出错（如登录、下单、审批失败），无明显临时绕过手段 | 高优先修复；不修复不能交付 |
| Major | M | 主要缺陷：功能可用但行为偏离规范（如分页越界返回异常），有临时绕过 | 计划内修复；如时间不足可写入《测试总结》"已知问题" |
| Minor | N | 次要缺陷：体验或边界场景问题（如错误提示文案不准） | 视时间安排，可不阻塞交付 |
| Trivial | T | 琐碎问题：无业务影响（如日志格式、注释错别字） | 不影响交付，记录归档 |

> 在《测试总结》中需按上述五级分别统计本次测试发现的缺陷数。

## 9 暂停准则和继续准则

### 9.1 暂停准则

满足下列任一条件即暂停后续测试：

- 被测系统**无法编译**或核心模块（如 `UserServiceImpl`、`OrderServiceImpl`）出现 **Blocker 级**缺陷，导致 ≥ 50% 用例无法执行；
- 测试基座（`AbstractControllerIntegrationTest`、`TestDataFactory`、H2 配置）失效，所有集成测试无法启动；
- 测试代码与被测代码**版本不一致**，可重复性不能保证；
- 关键依赖（JDK / Maven / H2）在所有可用机器上均不可用。

### 9.2 继续准则

恢复测试需同时满足：

- 暂停的根因问题已修复并通过开发者自测；
- 测试基座可正常初始化数据；
- 重新执行 `mvn clean test`，至少恢复到暂停前最近一次的通过率；
- 暂停期间引入的所有变更已合入测试基线。

## 附录 D JaCoCo 度量与 IEEE 829 覆盖术语映射

| IEEE 829 / 教材术语 | JaCoCo 度量名 | 含义 | 本计划采用 |
| --- | --- | --- | --- |
| 语句覆盖（Statement Coverage） | Instruction Coverage（指令覆盖） / Line Coverage（行覆盖） | JaCoCo 不直接给出"语句"维度；以指令覆盖率作为主报告口径，与"语句覆盖"近似一致 | 是 |
| 判定覆盖（Decision / Branch Coverage） | Branch Coverage（分支覆盖） | 每个分支真假各被执行一次 | 是 |
| 路径覆盖（Path Coverage） | 无原生支持 | JaCoCo 不直接度量；本项目不强制路径覆盖 | 否 |
| 方法覆盖 | Method Coverage | 每个方法是否被调用 | 仅作辅助参考 |
| 类覆盖 | Class Coverage | 每个类是否被加载 | 仅作辅助参考 |

> 因此本计划在 §7、§8 中说"语句覆盖率"即指 JaCoCo 的指令覆盖率（Instruction），"判定覆盖率"即指分支覆盖率（Branch）。在《测试总结》中也建议沿用此对应关系。
